---
name: feature-build
description: Execute a phase spec generated by /feature-plan. Pass a feature issue number to auto-pick the next unchecked phase, or pass a spec path directly. Branches off the feature branch, implements the phase, merges back into the feature branch, and updates the phase tracker on the feature issue.
model: sonnet
---

## Arguments

- `{issue-number|phase-spec-path}` — feature issue number (picks the first unchecked phase) or explicit path to a phase spec file (e.g. `.orc/features/storefront/phase-2.md`)
- `[--dry-run]` — optional; plan and score confidence without executing

---

## Steps

### 1. Parse arguments

Check whether the argument is a feature issue number (pure integer) or a spec path.

**If no argument was provided**, stop:
```
Which phase should I build? Pass a feature issue number or a spec path.
  /feature-build 42
  /feature-build .orc/features/storefront/phase-2.md
```

**If the argument is a feature issue number:**

Invoke the `issue-loader` agent. Pass the number and `{owner}/{repo}`. Confirm the returned labels include `feature`.

Derive `{slug}` from the returned `slug` field.

Read `.orc/features/{slug}/feature.json`:
```bash
cat .orc/features/{slug}/feature.json
```

If missing, stop:
```
feature.json not found in .orc/features/{slug}/.
Run /feature-plan {number} first.
```

Find the `## Phases` section in the issue body. Locate the first unchecked item (`- [ ]`) and extract its phase number `{n}`.

If no unchecked phases remain, stop:
```
All phases are complete for #{number} — {Name}.
Run /feature-merge {number} to merge into main.
```

Set `{phase-spec-path}` to `.orc/features/{slug}/phase-{n}.md`.

**If the argument is a spec path:**

Set `{phase-spec-path}` to the provided path.

Read `.orc/features/{slug}/feature.json` to get the feature issue number:
```bash
cat .orc/features/{slug}/feature.json
```

If missing, stop:
```
feature.json not found in .orc/features/{slug}/.
Run /feature-plan {number} first.
```

---

**Derive from `{phase-spec-path}` and `feature.json`:**
- `{slug}` — the feature directory name under `.orc/features/` (e.g. `storefront`)
- `{n}` — the phase number from the filename (e.g. `2` from `phase-2.md`)
- `{feature-issue}` — issue number from `feature.json`
- `{branch}` — `phase/{slug}/{n}` (e.g. `phase/storefront/2`)
- `{feature-branch}` — `feature/{slug}` (e.g. `feature/storefront`)
- `{worktree-path}` — `../{basename-of-pwd}-phase-{slug}-{n}` (e.g. `../myrepo-phase-storefront-2`)

If the spec file does not exist, stop:
```
Phase spec not found: {phase-spec-path}
Run /feature-plan {number} to generate it first.
```

---

### 2. Read spec

Read the full contents of `{phase-spec-path}`.

Parse the following sections:
- `## Overview` — one-paragraph summary
- `## Depends on` — list of prior phases required, or "None"
- `## Scope` — in scope / out of scope
- `## Codebase touchpoints` — files the implementer should know about
- `## Files to create` — new files and their purpose
- `## Files to modify` — existing files and what changes
- `## Data models / migrations` — schema changes if any
- `## Key logic` — non-obvious implementation details
- `## Acceptance criteria` — numbered verifiable outcomes

Display: phase name (from the `# Phase {n} — {name}` heading), overview, and acceptance criteria.

---

### 3. Enforce Depends On

If the `## Depends on` section says "None", skip this step.

Otherwise, for each prior phase listed, derive its expected branch name as `phase/{slug}/{prior-n}`.

Check whether that branch has been merged into `origin/feature/{slug}`:

```bash
git fetch origin
git branch -r --merged origin/{feature-branch} | grep "phase/{slug}/{prior-n}"
```

If any required phase branch is not merged, stop:
```
Phase {prior-n} must be merged before building phase {n}.

Missing: phase/{slug}/{prior-n} (not yet merged into {feature-branch})

Complete and merge that phase first, then re-run /feature-build {phase-spec-path}.
```

---

### 4. Branch + Worktree

Skip entirely if `--dry-run`.

```bash
git branch --list "{branch}"
git worktree list | awk -v b="[{branch}]" '$3 == b {print $1}'
```

**Branch does not exist — fresh build:**
```bash
git fetch origin
git worktree add {worktree-path} -b {branch} origin/{feature-branch}
```

**Branch exists, no worktree:**
```bash
git worktree add {worktree-path} {branch}
```

**Branch exists and worktree exists — prior partial attempt:**
```
Branch {branch} already exists at {worktree-path}.

(r) resume from where it left off
(s) hard reset to origin/{feature-branch} — discards all changes on this branch
```
On `(s)`: `git -C {worktree-path} reset --hard origin/{feature-branch}`

**Continue all remaining steps from within `{worktree-path}`.** All file reads, writes, and git commands run from there.

---

### 5. Approach summary

Read the spec's **Codebase Touchpoints**, **Files to Create**, and **Files to Modify** sections. Summarize as 3–5 bullets confirming what will change and why.

```
Approach:
- {bullet}
...
```

If more than 20 files appear across those sections, flag this before proceeding.

---

### 6. Confidence score

Score the approach 0–100 based on:
- Spec completeness (are acceptance criteria concrete and verifiable?)
- Codebase clarity (do touchpoint files exist and match expectations?)
- Absence of ambiguity or unknown dependencies

**Score ≥ 70 — auto-proceed:**
```
Confidence: {score}/100 — {brief reason}
```
Proceed to execution immediately.

**Score < 70 — human review required:**
```
Confidence: {score}/100 — {brief reason}

Say "proceed" when ready to build.
```
Wait for the user. Discuss or revise as needed. On "proceed", continue.

**If `--dry-run`:** stop here regardless of score. Report the score and approach.

---

### 7. Execute

Update status to signal a build is underway:
```bash
gh issue edit {feature-issue} --repo {owner}/{repo} \
  --remove-label "status:ready,status:built" \
  --add-label "status:progress"
```

Work through the spec in this order:
1. Data models / migrations
2. Files to create
3. Files to modify
4. Key logic (woven into the above as applicable)

Do not stop between steps unless blocked.

**No guessing allowed.** If a step is ambiguous, conflicts with the codebase, or the spec is silent on a required detail — stop and ask:
- Which step triggered the question
- What is ambiguous or missing
- What options exist and which you lean toward

---

### 8. Verify

Run commands listed in `AGENTS.md` under `## Verification`. Skip if the section does not exist.

---

### 9. Pre-commit gate

Run commands listed in `AGENTS.md` under `## Pre-commit Gate`. If a command fails: stop and report which command failed and why. Do not proceed until resolved.

If a command cannot run (missing binary, wrong environment): record it and continue.

---

### 10. Commit and push

Review changes before staging:
```bash
git status --short
git diff --stat
```

Identify any unrelated changes. If found:
```
{n} unrelated change(s) found:
  {file} — {description}

Commit these separately first?
(y) yes  (n) leave them unstaged
```

Stage relevant files explicitly — do not use `git add .`.

Commit with a conventional message derived from the phase overview:
```
feat(phase/{slug}/{n}): {one-line summary from Overview}
```

```bash
git push -u origin {branch}
```

---

### 11. Open PR

Derive `{owner}/{repo}` from the remote:
```bash
gh repo view --json nameWithOwner --jq '.nameWithOwner'
```

Open a pull request targeting the feature branch (not main):
```bash
gh pr create \
  --title "feat(phase/{slug}/{n}): {one-line summary from Overview}" \
  --body "$(cat <<'EOF'
## Phase {n} — {name}

{Overview paragraph from spec}

## Acceptance criteria
{numbered list from spec}

Part of #{feature-issue} — {feature-name}
EOF
)" \
  --base {feature-branch} \
  --head {branch}
```

Capture the PR number from the output.

---

### 12. CI check

```bash
gh pr checks {pr} --repo {owner}/{repo} --watch
```

If all checks pass: proceed.

If any checks are failing, enter CI fix mode:

1. Fetch logs for the first failing check:
   ```bash
   gh run view {run-id} --repo {owner}/{repo} --log-failed
   ```
2. Get the branch diff:
   ```bash
   git diff origin/{feature-branch}...HEAD
   ```
3. Invoke the `ci-debugger` agent. Pass the failing check name, log output, and diff.
4. Evaluate the proposed fix:
   - **Apply silently** if confidence is high and the fix is clearly safe (e.g. missing import, type error, lint issue).
   - **Stop and report** if confidence is low or the fix touches logic that could contradict the spec.
5. Apply, commit (`fix(phase/{slug}/{n}): fix CI — {brief description}`), push.
6. Re-watch CI and repeat until all checks pass.

---

### 13. Merge into feature branch

Squash-merge the PR, delete the phase branch, and remove the worktree:

```bash
MAIN=$(git worktree list | awk 'NR==1 {print $1}')

gh pr merge {pr} --repo {owner}/{repo} --squash
git -C "$MAIN" branch -D {branch}
git -C "$MAIN" fetch origin
git -C "$MAIN" worktree remove --force {worktree-path}
```

---

### 14. Update phase tracker

Fetch the current feature issue body:
```bash
gh issue view {feature-issue} --repo {owner}/{repo} --json body --jq '.body'
```

In the `## Phases` section, find the line for phase {n} and check it off:
```
- [ ] Phase {n} — {name}
```
→
```
- [x] Phase {n} — {name} (merged #{pr})
```

Update the issue:
```bash
gh issue edit {feature-issue} --repo {owner}/{repo} --body "{updated-body}"
```

Determine `{next-n}` — the next unchecked phase in the `## Phases` list, if any.

If all phases are now checked (`- [x]`), update status:
```bash
gh issue edit {feature-issue} --repo {owner}/{repo} \
  --remove-label "status:progress" \
  --add-label "status:built"
```

Otherwise, update status back to ready for the next phase:
```bash
gh issue edit {feature-issue} --repo {owner}/{repo} \
  --remove-label "status:progress" \
  --add-label "status:ready"
```

---

### 15. Report

```
Phase {n} merged — {name}
PR:    #{pr} (squash-merged into {feature-branch})
Tracker updated on #{feature-issue}.
```

If there is a next phase:
```
Run /feature-build {feature-issue} to continue with phase {next-n}.
```

If all phases are now checked:
```
All phases complete — status:built.
Run /feature-merge {feature-issue} to merge into main.
```
